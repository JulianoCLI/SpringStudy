Fase 1

Ambiente e Dependências

0/4
  • Analisar os arquivos e classes atuais do projeto no seu IDE (compreender a estrutura geral).
    O que faz?

    Entender a estrutura básica do projeto nos permite enxergar onde as responsabilidades estão divididas. Veja como LivrosModel possui uma referência direta para UserModel, indicando uma relação bidirecional.

    Onde pesquisar?

  • Verificar o arquivo pom.xml e garantir que a dependência do banco de dados H2 esteja incluída.
    O que faz?

    O H2 Database é um banco de dados relacional que pode rodar em memória (dentro da RAM). Ele é perfeito para estudo pois dispensa a necessidade de instalar servidores de bancos locais como MySQL ou PostgreSQL.

    Como fazer?

    Abra o pom.xml, procure pela tag <dependencies> e verifique se há uma dependência com o artifactId h2. Caso não haja, você pode declará-la com o groupId com.h2database.

    Onde pesquisar?

    H2 Database Spring Boot Dependency na internet.

  • Configurar as propriedades do banco H2 no application.properties para habilitar o console de testes.
    O que faz?

    O console H2 é um painel web visual que roda junto com a aplicação. Ele permite consultar as tabelas criadas automaticamente pelo Hibernate e testar se os dados estão sendo salvos de verdade.

    Como fazer?

    Adicione chaves no seu arquivo application.properties para habilitar o console e configurar um datasource de testes (ex: spring.h2.console.enabled=true e spring.datasource.url=jdbc:h2:mem:testdb).

    Onde pesquisar?

    How to enable h2-console in Spring Boot.

  • Rodar o projeto no terminal (mvnw spring-boot:run) e verificar se inicializa corretamente.
    O que faz?

    O Maven Wrapper (mvnw) compila o código, baixa as dependências necessárias e sobe o contêiner interno do Tomcat na porta local padrão 8080.

    Como fazer?

    No prompt de comando ou bash na raiz do projeto, execute ./mvnw spring-boot:run e leia o terminal buscando erros em vermelho. Se inicializar sem falhas, sua base está sólida!

    Onde pesquisar?

    Spring Boot Maven Plugin spring-boot:run target.

Fase 2

Ajustes de Modelo (Lombok & JPA)

0/8
  • Abrir a classe LivrosModel.java e entender o relacionamento `@ManyToOne` com o UserModel.
    O que faz?

    A anotação @ManyToOne expressa que muitos livros podem estar em posse de uma única pessoa (usuário). O @JoinColumn(name = "user_id") mapeia a coluna física de chave estrangeira que armazena a relação.

    Como fazer?

    Observe que o campo UserModel user é o que faz o elo. Se o valor for null na base, o livro está livre para empréstimo.

    Onde pesquisar?

    JPA ManyToOne and JoinColumn annotations.

  • Adicionar a anotação @EqualsAndHashCode(onlyExplicitlyIncluded = true) na classe LivrosModel.
    O que faz?

    Por padrão, o Lombok cria métodos equals/hashCode avaliando todos os campos da entidade. No JPA, isso pode gerar loops infinitos nas coleções devido a relações bidirecionais de banco. Ao definirmos onlyExplicitlyIncluded = true, avisamos ao Lombok para comparar os objetos apenas pelos atributos que marcarmos explicitamente.

    Como fazer?

    Coloque a anotação @EqualsAndHashCode(onlyExplicitlyIncluded = true) no topo da declaração da classe, junto com `@Getter` e `@Setter`.

    Onde pesquisar?

    Lombok @EqualsAndHashCode(onlyExplicitlyIncluded = true) for JPA entities.

  • Marcar o atributo id em LivrosModel com a anotação @EqualsAndHashCode.Include.
    O que faz?

    Isso especifica que o campo id é o único critério de diferenciação. Dois livros serão iguais na memória da JVM se, e somente se, tiverem o mesmo ID de banco de dados.

    Como fazer?

    Adicione @EqualsAndHashCode.Include logo acima do campo private long id; na classe LivrosModel.

    Onde pesquisar?

    Lombok @EqualsAndHashCode.Include usage guide.

  • Verificar os construtores de LivrosModel, garantindo o construtor vazio (`@NoArgsConstructor`).
    O que faz?

    O Hibernate necessita de um construtor padrão (sem argumentos) público ou protegido para instanciar os objetos dinamicamente quando executa as buscas SQL.

    Como fazer?

    Use a anotação do Lombok @NoArgsConstructor acima da classe. Certifique-se de que não conflita com construtores manuais pré-existentes.

    Onde pesquisar?

    Lombok @NoArgsConstructor and JPA entity requirement.

  • Abrir a classe UserModel.java e entender a relação bidirecional `@OneToMany` com os livros.
    O que faz?

    A anotação @OneToMany(mappedBy = "user") mapeia o lado oposto da relação. O parâmetro mappedBy diz ao Spring Boot que o atributo responsável pela chave estrangeira está mapeado do outro lado sob o nome 'user' na classe LivrosModel.

    Como fazer?

    Apenas verifique a declaração: private List<LivrosModel> livros. Ela nos dá a conveniência de ler os livros de um usuário diretamente da classe User.

    Onde pesquisar?

    JPA OneToMany mappedBy parameter.

  • Adicionar a anotação @EqualsAndHashCode(onlyExplicitlyIncluded = true) na classe UserModel.
    O que faz?

    Aplica o mesmo tratamento de isolamento de comparação no UserModel. O objetivo principal é impedir a inclusão da lista recursiva de livros na comparação, o que causaria um estouro de pilha (StackOverflowError) quando o equals fosse chamado.

    Como fazer?

    Insira a anotação @EqualsAndHashCode(onlyExplicitlyIncluded = true) sobre a classe UserModel.

    Onde pesquisar?

    Lombok equals and hashCode StackOverflow JPA relationship.

  • Marcar o atributo id em UserModel com a anotação @EqualsAndHashCode.Include.
    O que faz?

    Garante que dois usuários em memória que possuam o mesmo ID de banco sejam avaliados logicamente como a mesma entidade no Java.

    Como fazer?

    Coloque a anotação @EqualsAndHashCode.Include acima do campo private long id; em UserModel.

    Onde pesquisar?

    Lombok explicit field hashcode java.

  • Verificar se a lista livros está inicializada corretamente em UserModel.
    O que faz?

    Inicializar a lista evita o terrível erro de NullPointerException quando tentamos adicionar livros na coleção sem que o Hibernate tenha carregado ela previamente do banco de dados.

    Como fazer?

    Verifique se ela está declarada como: private List<LivrosModel> livros = new ArrayList<>();.

    Onde pesquisar?

    Best practices initializing collection in JPA entities.

    O que faz?

    O JpaRepository fornece as operações básicas de banco prontas. Ao estender, você ganha acesso aos métodos do banco para salvar, consultar, deletar e paginar sem precisar escrever uma linha de SQL.

    Como fazer?

    Onde pesquisar?

    Spring Data JpaRepository generics parameters.

  • Abrir a interface UserRepository.java e certificar que ela estende JpaRepository<UserModel, Long>.
    O que faz?

    Disponibiliza o repositório necessário para ler e persistir dados cadastrais na tabela tb_cadastro.

    Como fazer?

    Confirme a declaração: public interface UserRepository extends JpaRepository<UserModel, Long>.

    Onde pesquisar?

    Spring Data JPA UserRepository interface definition.

  • Estudar os métodos herdados do `JpaRepository` (ex: `save`, `findById`, `findAll`).
    O que faz?

    Os métodos mais importantes são save(entidade) (que insere registros se o ID for nulo ou atualiza se o ID já existir) e findById(id) (que busca um registro específico e retorna envolvido em um `Optional` para tratamento seguro de valores nulos).

    Como fazer?

    Compreenda como usar findById(id).orElseThrow(...). Esse encadeamento é a forma mais idiomática de obter o valor e lançar uma exceção de tratamento de erro no mesmo passo.

    Onde pesquisar?

    Java Optional orElseThrow and JpaRepository findById usage.

  • Garantir que ambos os arquivos possuem a anotação @Repository.
    O que faz?

    A anotação @Repository é um estereótipo do Spring que instrui o contêiner de inversão de controle (IoC) a injetar essa classe como um bean do sistema e a traduzir erros nativos do banco em exceções legíveis do ecossistema do Spring.

    Como fazer?

    Adicione @Repository antes da declaração da assinatura de suas interfaces.

    Onde pesquisar?

    Spring @Repository annotation purpose.

Fase 4

Logica de Empréstimo (Service)

0/9
  • Criar a assinatura do método no BibliotecaService.java: public void realizarEmprestimo(Long idUsuario, Long idLivro).
    O que faz?

    Esta assinatura define o canal pelo qual a API solicitará o início de um processo de empréstimo. Ela recebe o id do usuário que quer ler e o id do livro que ele quer retirar.

    Como fazer?

    Declare o método público sem retorno (void) e com parâmetros correspondentes.

    Onde pesquisar?

    Spring Service layer rules.

  • Injetar via construtor as dependências do LivrosRepository e UserRepository dentro de `BibliotecaService`.
    O que faz?

    A injeção de dependência por construtor é a melhor prática no Spring. Ela permite declarar as propriedades como private final, garantindo imutabilidade e facilitando a escrita de testes unitários que não necessitam subir todo o contexto do banco de dados.

    Como fazer?

    Declare os campos dos repositórios como private final na classe e crie um construtor que inicialize ambos. Se preferir automatizar, coloque a anotação @RequiredArgsConstructor do Lombok sobre a classe BibliotecaService.

    Onde pesquisar?

    Spring Boot Constructor-based Dependency Injection vs @Autowired.

  • Adicionar a anotação @Transactional sobre o método para garantir a atomicidade.
    O que faz?

    A anotação @Transactional gerencia a transação com o banco de dados. Caso qualquer exceção de negócio seja disparada durante o fluxo (ex: limite excedido), todas as modificações que foram enviadas para as tabelas são imediatamente desfeitas (Rollback), mantendo os dados seguros.

    Como fazer?

    Escreva @Transactional acima da declaração do método de empréstimo ou devolução. Importe o pacote org.springframework.transaction.annotation.Transactional.

    Onde pesquisar?

    Spring Framework @Transactional annotation rollback rules.

  • Buscar o usuário pelo ID e lançar exceção se ele não existir.
    O que faz?

    Evita prosseguir com requisições que possuam referências inválidas a usuários fictícios.

    Como fazer?

    Chame userRepository.findById(idUsuario) e use o método .orElseThrow() para disparar uma exceção de falta de recurso (pode ser um generic `RuntimeException` inicialmente ou a sua exceção customizada da Fase 8).

    Onde pesquisar?

    Optional findById orElseThrow in Spring Boot.

  • Buscar o livro pelo ID e lançar exceção se ele não existir.
    O que faz?

    Busca a representação física do livro do banco de dados para podermos ler o estado da propriedade `user` e validar se está ou não na prateleira.

    Como fazer?

    Chame livrosRepository.findById(idLivro).orElseThrow(() -> new RuntimeException("Livro não encontrado")).

    Onde pesquisar?

    Spring Data findById logic flow.

  • Validar a disponibilidade do livro: verificar se livro.getUser() != null.
    O que faz?

    O livro está livre se, e somente se, a coluna user_id (atributo `user`) for null. Se houver um usuário preenchido, o livro já está emprestado e não podemos permitir um novo empréstimo duplicado.

    Como fazer?

    Crie um condicional if: se livro.getUser() != null, dispare um erro informando que o livro já está indisponível para empréstimo.

    Onde pesquisar?

    JPA Entity state validation in business logic.

  • Validar o limite do usuário: verificar se o tamanho da lista usuario.getLivros().size() >= 3.
    O que faz?

    Verifica se o usuário já atingiu a cota máxima permitida pela regra de negócios (no máximo 3 livros sob posse concomitante).

    Como fazer?

    Obtenha a coleção através de usuario.getLivros(), leia o tamanho usando o método .size() e, caso ele seja maior ou igual a 3, bloqueie a operação lançando um erro de negócio.

    Onde pesquisar?

    JPA lazy collections size checking.

  • Executar a ação: fazer a associação definindo o usuário no livro (livro.setUser(usuario)).
    O que faz?

    Isso estabelece o elo de banco em memória de dados. Como o JPA trabalha de forma orientada a objetos, setar o UserModel no objeto do livro altera o estado gerenciado da entidade.

    Como fazer?

    Chame o método modificador: livro.setUser(usuario).

    Onde pesquisar?

    JPA Bi-directional relationships setter best practices.

  • Salvar a alteração no banco de dados usando o método do repositório livrosRepository.save(livro).
    O que faz?

    Comunica as alterações da entidade com o banco de dados. O Hibernate gerará o comando UPDATE tb_livros SET user_id = ? WHERE id = ? internamente.

    Como fazer?

    Execute livrosRepository.save(livro) no final do escopo do método.

    Onde pesquisar?

    Spring Data JpaRepository save method behavior.

Fase 8

Tratamento Global de Exceções

0/6
  • Criar a classe de exceção customizada de negócio (ex: RegraDeNegocioException).
    O que faz?

    Criar exceções específicas nos permite interceptar erros de negócio e devolvê-los ao usuário com um código HTTP adequado (ex: 400 Bad Request) ao invés de estourar um erro 500 (Internal Server Error) feio no navegador.

    Como fazer?

    Crie uma classe pública chamada RegraDeNegocioException que estenda de RuntimeException. Implemente um construtor que receba uma mensagem (super(message)).

    Onde pesquisar?

    Custom RuntimeException Java Spring Boot.

  • Criar uma exceção de recurso inexistente (ex: RecursoNaoEncontradoException).
    O que faz?

    Sinaliza especificamente quando uma busca de repositório falha ao encontrar registros por um ID.

    Como fazer?

    Crie a classe estendendo de RuntimeException para podermos responder com o status de erro 404 (Not Found).

    Onde pesquisar?

    Spring Boot ResourceNotFoundException custom class.

  • Atualizar a camada de serviço (`BibliotecaService`) para lançar estas novas exceções.
    O que faz?

    Garante que a service emite alertas específicos e tipados em caso de falhas nas verificações de negócio.

    Como fazer?

    Substitua os antigos lançamentos de erros (como new RuntimeException()) no BibliotecaService por new RegraDeNegocioException("...") ou new RecursoNaoEncontradoException("...").

    Onde pesquisar?

    Throw custom exceptions Java Spring Boot best practices.

  • Criar uma representação padronizada do corpo de erro (ex: classe de retorno JSON).
    O que faz?

    Padroniza o retorno dos erros da API em uma resposta estruturada de dados, facilitando a interpretação do front-end ou do cliente HTTP.

    Como fazer?

    Crie uma classe Java simples de modelo (ex: StandardError) que tenha atributos de mensagem, código HTTP e timestamp da ocorrência.

    Onde pesquisar?

    Standard error message JSON payload API design.

  • Adicionar o método `@ExceptionHandler(RegraDeNegocioException.class)` no GlobalExceptionHandler para retornar HTTP 400.
    O que faz?

    A anotação @RestControllerAdvice (presente na classe do handler) faz com que ela escute e intercepte exceções lançadas em qualquer controller da aplicação. O @ExceptionHandler(MinhaClasse.class) indica qual método tratará qual erro.

    Como fazer?

    Crie um método anotado com @ExceptionHandler(RegraDeNegocioException.class) que monte a sua representação de erro e retorne com o status HttpStatus.BAD_REQUEST.

    Onde pesquisar?

    Spring Boot @RestControllerAdvice and @ExceptionHandler tutorial.

  • Adicionar o método `@ExceptionHandler(RecursoNaoEncontradoException.class)` para retornar status HTTP 404.
    O que faz?

    Mapeia a exceção de recursos inexistentes, devolvendo o erro 404 (Not Found) quando um livro ou usuário de ID inexistente for consultado.

    Como fazer?

    Crie o método no handler com a anotação @ExceptionHandler(RecursoNaoEncontradoException.class) retornando status HttpStatus.NOT_FOUND.

    Onde pesquisar?

    Spring HttpStatus.NOT_FOUND @ResponseStatus handler.
